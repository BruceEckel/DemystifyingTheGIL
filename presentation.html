The GIL

Race conditions, free-threaded Python, and why your threads lied to you.

What is the GIL?

  • The Global Interpreter Lock is a mutex inside CPython
  • Only one thread executes Python bytecode at a time
  • Protects CPython's internal reference counts from corruption
  • Makes single-threaded code faster — no per-object locking needed
  • Side effect: accidentally hides race conditions in your code
The GIL is not a feature of Python the language — it's an implementation detail of CPython.

counter += 1 is not atomic

One innocent line compiles to three bytecode operations:

LOAD read counter from memory
ADD compute counter + 1
STORE write result back to counter
A thread switch can happen between any two of these steps.
Two threads read the same value → both increment it → one write is lost.

unsafe.py — The Race Condition

# 8 threads, each incrementing 100,000 times
counter = 0

def increment(iterations):
    global counter
    for _ in range(iterations):
        counter += 1  # ← not atomic!

# Expected: 800,000
# With GIL:  800,000  ✓  (GIL serializes the 3 bytecodes)
# No GIL:    ???      ✗  (race condition exposed)

Run it:

$ python unsafe.py              # with GIL
$ uv run --python 3.14t unsafe.py  # without GIL

What You'll See

Standard Python 3.14 (GIL active)

Python 3.14: Standard GIL
Expected: 800,000
Actual:   800,000

GIL serialises the bytecodes — race never fires.

Python 3.14t (GIL removed)

Python 3.14t: No GIL
Expected: 800,000
Actual:   614,332

Real parallelism exposes the bug.

safe.py — Use a Lock

lock = threading.Lock()

def increment(iterations):
    global counter
    for _ in range(iterations):
        with lock:        # ← protect the non-atomic operation
            counter += 1
Python 3.14: Standard GIL
Actual: 800,000 ✓
Python 3.14t: No GIL
Actual: 800,000 ✓
Correct on both builds. Don't rely on the GIL for thread safety — use explicit synchronisation.

Python 3.14t — Free-Threaded

  • Experimental build: install the t variant (3.14t) via uv or pyenv
  • GIL is disabled by default — threads run in true parallel on multiple cores
  • Exposes latent race conditions that the GIL was silently hiding
  • PEP 703 — "Making the Global Interpreter Lock Optional"
$ uv run --python 3.14t script.py

# Detect in code:
import sys
free_threading = "free-threading" in sys.version

Key Takeaways

  • The GIL serialises bytecode execution — it's not a deliberate thread-safety guarantee
  • counter += 1 is a read-modify-write — never safe without a lock
  • Code that "works" with threads under CPython may be silently broken
  • Python 3.14t reveals those bugs — use it as a correctness checker
  • Always protect shared mutable state with threading.Lock, queue.Queue, or similar primitives
The rule: if two threads touch the same data and at least one writes, you need a lock — GIL or not.
