Act 2

What Just Happened?

Inside the GIL — and everything it was quietly doing for you

~10 minutes

What Is the GIL?

  • The Global Interpreter Lock — a mutex inside CPython
  • Only one thread executes Python bytecode at a time
  • Not a Python language feature — it's a CPython implementation detail
  • Other implementations (Jython, PyPy-STM) don't have it
  • Has existed since Python 1.5 (1997)
The GIL is not a deliberate thread-safety guarantee. It's a trade-off made for simplicity and single-threaded speed.

Everything the GIL Does

The GIL is doing far more work than most people realize:

  • Reference counting — ob_refcnt inc/dec must be atomic
  • Memory allocator — PyMem_Malloc is not thread-safe without it
  • Cyclic GC — needs exclusive traversal of the object graph
  • CPython internals — module dicts, type objects, interned strings
  • Import system — sys.modules integrity during import
  • Signal handling — main thread receives signals via GIL handoff
  • C extensions — written assuming single-threaded bytecode execution
  • sys.settrace / profiling — frame inspection assumes serialization
  • Accidental thread safety — your code works even without locks

counter += 1 Is Not Atomic

One innocent line compiles to three bytecode operations:

LOAD read counter from memory
↓
ADD compute counter + 1
↓
STORE write result back to counter
A thread switch can happen between any two of these steps.
Two threads read the same value → both increment → one write is lost.

Why It "Works" With the GIL

  • The GIL is released at regular intervals — every N bytecodes (the "check interval")
  • Those 3 bytecodes are short enough that the GIL often isn't released between them
  • But this is not guaranteed — it's timing luck
  • Even with the GIL, long enough critical sections can be interrupted
You've been getting away with it — not writing correct code.

The Fix: Explicit Locking

lock = threading.Lock()

def increment(iterations):
    global counter
    for _ in range(iterations):
        with lock:        # ← protect the non-atomic operation
            counter += 1

Actual: 800,000 ✓
Python 3.14t (no GIL)
Actual: 800,000 ✓
The lock makes the intent explicit — don't rely on interpreter accidents.

Python 3.14t — Free-Threaded Build

  • PEP 703 — "Making the Global Interpreter Lock Optional" (Sam Gross, 2022)
  • Experimental since Python 3.13 — install the t variant alongside standard Python
  • GIL disabled by default — threads run in true parallel on multiple cores
# Detect in code:
import sys
free_threading = "free-threading" in sys.version
# Install and run:
$ uv python install 3.14t
$ uv run --python 3.14t script.py
Act 3

The Coming Surprise

Why switching builds will break code you didn't write

~10 minutes

The Iceberg

You rewrote your code. You added locks. Your tests pass under 3.14t.

But you didn't write most of the code you run.

Your application code — the part you can see and fix
~ waterline — the GIL boundary ~
  • Every library you import (requests, SQLAlchemy, numpy, …)
  • Every C extension you installed
  • The standard library itself
  • CPython's own object model and allocator

Most library authors never needed to think about thread safety. The GIL was their lock — silently, invisibly, for free.

Library Patterns That Will Break

Code that is "accidentally thread-safe" under CPython today:

  • Module-level mutable state — shared caches, counters, registries initialized at import time
  • Lazy initialization — if _cache is None: _cache = build() (classic TOCTOU race)
  • dict / list mutations — appending to a shared list, updating a shared dict
  • Connection pools — checkout/checkin logic that assumes serialized access
  • Logging handlers — writing to shared buffers or files
  • C extensions — any extension that touches Python objects without holding the GIL

A Concrete Example

A pattern found throughout library code — looks safe, isn't:

# module-level shared state — common in caching, registries, plugins
_registry = {}

def register(name, obj):
    _registry[name] = obj   # dict write — "safe enough" under GIL

def lookup(name):
    return _registry.get(name)

Under 3.14t with concurrent register() calls:

  • Two threads resize the dict simultaneously → internal structure corruption
  • One thread iterates while another inserts → RuntimeError: dictionary changed size
The code didn't change. The behavior did.

Why This Is Hard to Find

  • Race conditions are non-deterministic — the bug may appear 1 in 10,000 runs
  • Test suites run sequentially or with low concurrency → CI stays green
  • The bug surfaces under production load, specific hardware, or after an OS scheduler change
  • Python 3.14t makes races more likely to fire — but still not certain
Luckily: 3.14t is a free correctness checker you can run against your test suite today.

What You Should Do Now

  1. Audit shared mutable state — anything touched by more than one thread
  2. Run your test suite under 3.14t — it exposes latent races the GIL was hiding
  3. Add explicit locks — threading.Lock, threading.RLock, queue.Queue
  4. Prefer immutable data — objects created once and never mutated are always safe
  5. Check your dependencies — file issues against libraries that don't declare thread safety

The Payoff

Why bother? Because without the GIL:

  • CPU-bound threads actually run in parallel on multiple cores
  • threading becomes a genuine parallelism tool, not just an I/O concurrency tool
  • Python can compete with Go and Java for multi-core throughput
  • The ecosystem has 30 years of battle-tested concurrency primitives to build on
  • Libraries that do the work correctly will be faster and more scalable
The era of the GIL as a crutch is ending. The era of real Python parallelism is beginning.

The Rule

If two threads touch the same data
and at least one of them writes,
you need a lock — GIL or not.

  • This has always been true
  • The GIL made it optional in CPython — that era is ending
  • Code that follows this rule is correct on every Python implementation, today and tomorrow
